# Delivery Assessment
For this sprint, I was tasked with stabilizing the testing and linting workflows for the OSS DigitalBonesBox project. While I was initially tasked with writing unit, integration, static and security tests, there were persistent Jest test failures for this project which I believe needed to be resolved first before I began writing any kind of tests. Thus, I had to ensure that CI/CD workflows could execute successfully without affecting production code.

After setting up the Jest testing framework in the previous sprint, I began the current sprint by reproducing the failing Jest tests locally and incrementally logging DOM states to understand why tests were failing. By conducting this process, I understood that the errors were a result of jsdom environment limitations as opposed to the project's logic/pre-existing code. Specifically, jsdom automatically normalized empty image src values to http://localhost/ and event listeners bound on DOMContentLoaded were not executed during Jestâ€™s runtime. Through a process of validation by running controlled tests, I created a lightweight jest.setup.js file to simulate DOMContentLoaded events and prevent environment-based failures. I also made sure to stabilize the project's ESLint configuration by migrating from the deprecated .eslintignore to the new flat configuration format within eslint.config.js, which allowed the linting errors to be resolved smoothly and the project passed the command "npm run lint" successfully. Overall, even though I wasn't able to write the required tests, my deliverables were validated locally and through CI, and my pull request was merged successfully without affecting the application's runtime behavior. These updates improved code reliability and maintainability, which strengthened the project's overall infrastructure.
